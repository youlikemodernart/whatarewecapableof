Systems research

WAWCO is researching how AI systems can preserve source material, coordinate agent work, route candidate state through review, and keep human judgment at the point where work becomes accepted. This page maps the active research: current abstractions, research arms, evidence streams, and operating rules.

This is research, planning, fake-data implementation, and internal review. Live data and external writes require explicit approval. Markdown and structured files are source truth; this HTML page is a review surface generated from those files.

From ethos to operating system

The guiding reference is a fictional text about a project environment where accumulated material becomes active context. Documents, stakeholder claims, design proposals, constraints, and decisions compose into a namespace that the project's participants can navigate. The system's purpose is to let accumulated work remember what it is.

Four principles govern the translation into operating patterns. Source material remains active: research, interviews, notes, and stakeholder comments continue to act on future proposals. Every claim carries provenance: recommendations and generated work point back to the material that produced them. Candidate work moves through review: new proposals surface relevant constraints, conflicts, and prior decisions. Humans accept or reject state: the system provides memory, consequence, and surface area for judgment, while people compose and decide.

Memory works through graduated commitment. Most context stays light: session notes, workboards, source packets, handoffs, and closeouts. Durable memory holds context that is source-backed, useful across sessions, and accepted. Context moves through stages: raw event, observation, candidate lesson, repeated pattern, operating rule, stable principle.

Core abstractions

Skills

Methods and procedural references. A skill defines how to do a class of work: procedures, reference files, constraints, and routing tables. Skills live in ~/.pi/agent/skills/ and are shared between Pi and Claude Code through symlinks.

Examples: web-design, writing-style, orchestration-workboard, mail-intake

Agents

Executable roles. An agent is a role prompt that can perform work when routed with bounded prompts and source packets. Shared agents exist in both Pi format and Claude Code format so the role can travel across harnesses.

Examples: consulting-pipeline, web-design, ezra-production, typography

Talent manifests

Descriptive capability contracts. A talent manifest sits above skills and agents. It describes how a role should be deployed: skills, memory roots, tools, approval gates, smoke tests, known risks, and harness support. Talent manifests guide routing and review; they do not grant permissions.

Current shared test: consulting-pipeline

Workboards

Run coordination artifacts. A workboard tracks the state of a multi-step task: goals, source packets, task graphs, worker slots, routing decisions, conflicts, verifier notes, proposed durable updates, and closeout. Workboards coordinate process. They do not approve action.

orchestration-workboard skill, production-operations-mesa-pilot workboard

Proposal packets

Acceptance gates. A proposal packet evaluates candidate state before it becomes accepted, staff-facing, external, durable, pushed, deployed, remembered, or used as operational guidance. Approval intent recorded in a packet is still only intent. The parent session verifies state before acting.

orchestration-workboard / references / 04-project-proposal-packets.md

Responsibility packets

Recurring work units with explicit ownership. Each packet describes one job: current owner, candidate AI owner, inputs, outputs, source of truth, allowed actions, forbidden actions, approval gates, validation type, escalation path, reassignment options, and automation status.

responsibility-map-draft.md

Source ladders

Authority maps for conflicting systems. When several systems hold overlapping data, a source ladder establishes which system is authoritative for each field and action. If two sources conflict, the workflow produces an exception or proposal instead of choosing silently.

Project memory and pattern library

Two layers of accepted durable context. Project memory preserves facts, decisions, current state, and next actions for a specific project. The shared pattern library holds reusable patterns promoted from project pilots after client-specific data is stripped and evidence links are attached.

Review surfaces

HTML pages, preview deployments, dashboards, Discord summaries, and staff-visible drafts generated from source truth. Review surfaces make structured data legible for humans who do not read Markdown or YAML. Source truth stays in Markdown, CSV, JSON, YAML, workboards, proposal packets, and scripts.

Workspace substrates

The storage and versioning layers where project artifacts live. Current substrate: local Markdown files and Git. Mesa is under fake-data evaluation as a hosted versioning layer with proposal branches, diffs, merges, and a virtual filesystem. Cabinet has been studied for patterns but is not adopted as a live substrate.

Active research arms

Ethos and adaptive memory

How project context accumulates, stays active, and decays gracefully. Current work includes graduated commitment from raw event through candidate lesson to stable principle, witness chains from source to implication, and observation-before-action as the default agent posture. The mail-intake workflow is the first concrete observation-ledger pattern in use.

Orchestration and delegation

How parent sessions, subagents, workers, verifiers, and reviewers coordinate without losing source provenance or approval boundaries. Seven arXiv papers were synthesized in May 2026 on talent packaging, workboards, agent-native project artifacts, and verified memory evolution. The current posture is workboard-first: use thin coordination for active runs and defer broader automation until patterns prove useful across real pilots.

Model routing and tool governance

Which model tier handles which task, and how new tools enter the system. Use high reasoning for source authority, approval scope, contradictions, and operational policy. Downshift when the task is bounded, reversible, mechanical, and has a clear verifier. Every new online tool enters through an intake card before experimentation.

Current evidence stream: Ezra production operations

Ezra production operations is the first concrete pilot. The target is to map digital production-manager work into modular responsibilities that can move from manual to AI-assisted with human review and approval gates.

The pilot uses fake data exclusively. The first fake daily workflow produced a queue review, exception register, staff-visible review draft, source ladder, validation types, human approval matrix, model-routing observations, questions for staff, a fake spec proposal, a proposal packet, and a generated HTML review page.

The model-routing observation is still draft-level: field detection and register formatting look downshiftable after source-ladder rules are fixed. Source authority decisions, approval scope, transcript-to-spec proposals, and durable guidance remain high-reasoning work.

Candidate talents have been sketched for queue scouting, shipping exception triage, transcript-to-spec extraction, spec lookup, shop guide building, product data proposals, source ladder verification, and orchestration.

Open for this pilot: actual task inventory from the production team, Mesa workspace decision, staff-facing review surface usability, and which findings generalize into the shared pattern library.

Operating rules

  • Fake data first for new client-operation pilots.
  • Parent session remains single writer for live systems.
  • Agents scout, draft, verify, and review. They act on live systems only when a specific action is approved.
  • Proposal packets gate accepted state and external action.
  • Source truth stays Markdown, CSV, JSON, YAML, workboards, proposal packets, and scripts.
  • HTML is generated for human review and must be reversible.
  • Talent manifests describe boundaries. They do not grant permissions.
  • Workboards coordinate process. They do not approve action.
  • Approval intent is not application. Verification precedes action.
  • Project pilots feed the shared pattern library after client-specific data is stripped.

Open questions

  • Which abstractions stay lightweight files, and which deserve runtime tooling?
  • What belongs in project memory, shared pattern memory, a skill, an agent, or a talent manifest?
  • Which workspace substrate can safely hold proposal state for staff or client review?
  • What review surface is useful enough for nontechnical collaborators?
  • Which findings from current pilots generalize, and which stay project-specific?
  • How should context decay, supersession, and retirement work without losing useful continuity?
  • What is the smallest useful version of auto-projection in the current system?
