Tool intake process

Every new tool enters through a structured intake card before experimentation. The rubric covers data boundary, auth needs, source-truth model, artifact model, external write surface, export and deletion, security risks, Mesa fit, Ezra pilot fit, lowest-risk test, kill criteria, and decision. See the full rubric below.

Active decisions

  • Mesa stays fake-data or public-only until permissions, deletion, retention, export, recovery, and FUSE behavior are proven.
  • Parent session remains single writer for live systems.
  • Workboards coordinate active runs.
  • Proposal packets gate accepted state and external action.
  • Source truth stays Markdown, CSV, JSON, YAML, workboards, proposal packets, and scripts.
  • HTML is generated for human review and must be reversible: clean content recoverable as Markdown or structured data without reconstructing from layout markup.
  • Cabinet is a reference and pattern source. The current substrate is local Markdown, files, and Git.
  • Agents scout, draft, review, annotate, and verify. They act on live systems only when a specific action is approved.
  • Ezra-facing HTML requires the Ezra Arthur taste profile before review or sharing.
  • Ezra is a case study, not the architecture container. Reusable conclusions promote into the shared AI systems corpus with evidence links and without private or client-specific data.

Generalization pipeline

Project pilots are evidence sources. Before closeout or compaction, each lesson is classified as project-only context, reusable client-operations pattern, general AI systems principle, tool-evaluation finding, approval or security boundary, or candidate template.

Project-only context stays in the project folder. Reusable conclusions move into the shared pattern library with links back to the evidence source. Current promoted candidates include responsibility packets, source ladders, proposal gates, fake-data-first pilots, generated review surfaces, and tool intake cards.

Tool intake rubric

For every new tool Noah brings into a session, create a short intake card before experimenting. Process each tool through this rubric before using browser automation, cloning repos, installing packages, creating accounts, authenticating, or testing with project files.

Field Description
Tool name and URL The tool's name and primary address.
Category Workspace, agent runtime, knowledge base, browser automation, design artifact, data pipeline, workflow automation, model router, deployment, or other.
Claim What the tool says it solves.
Possible role Where the tool could fit in this architecture.
Data boundary Local, cloud, browser, external API, or unknown.
Auth and credentials What login, API keys, tokens, or permissions the tool needs.
Source-truth model Markdown, database, proprietary workspace, Git, files, API, or unknown.
Artifact model Generated, editable, embedded, hosted, or none.
External write surface What systems the tool writes to outside the local machine.
Export and deletion story Can data be exported cleanly? Can it be deleted completely?
Security and privacy risks Telemetry, data retention, third-party access, credential exposure, or other risks.
Mesa fit Compatibility with Mesa as a permissioned project workspace.
Ezra fake-pilot fit Compatibility with the Ezra production operations fake-data pilot.
Lowest-risk fake-data test The simplest test using only public or fake data.
Kill criteria Conditions that would disqualify the tool from further evaluation.
Decision Study, sandbox, borrow pattern, bypass, or propose deeper pilot.

Format policy

Future-agent-readable content stays Markdown or structured data. Repeatedly edited content stays Markdown or structured data. Long-lived content stays Markdown or structured data. HTML is generated for human review, dashboards, demos, prototypes, and stakeholder views.

Decision test for every artifact: who is the audience (humans, future agents, or both), what is the lifecycle (written once or edited repeatedly), and what is the horizon (day, quarter, or years).
