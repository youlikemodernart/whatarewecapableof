What are we capable of?
Prepared for PäSTE / April 2026

A first-build outline for PäSTE: a Shopify-connected collage interface, durable design state, automatic production packets, and a factory handoff simple enough to operate before the full marketplace is built.

Overview is what the first build should prove. System is the loose architecture. First Build is the MVP boundary, milestones, and questions to confirm with Davey and Andres.

Overview System First Build ?

PäSTE already has the right sketch: fans collage licensed IP onto garments, buy through Shopify, and the factory prints one unit at a time. The Cargo beta proves the interaction. The first MVP should prove the operating loop.

One paid line item should move from customer design to factory-ready packet without a screenshot, email reply, or manual recreation step. Once that works, the marketplace, affiliate layer, dashboards, and partner embeds have something real to attach to.

What the MVP proves

01

A customer can work inside an approved asset pack: drag, scale, rotate, layer, and place artwork on front and back garment templates.

02

The finished design is stored as a structured object: asset IDs, asset pack version, garment choice, side, position, scale, rotation, and layer order.

03

Shopify receives the design ID at checkout, keeps the product and payment flow clean, and triggers the next step when the order is paid.

04

The backend turns the paid order into a production packet with proof images, print files, product details, shipping details, and QC instructions.

05

The royalty ledger records who should be credited: IP holder, guest designer if present, affiliate if present, and PäSTE retained margin.

What remains open

The first build can leave several ideas in planning: partner-site embeds, automated payouts, AI-assisted collage, user uploads, a public resale marketplace, and multi-IP dashboards. Those layers become easier to design once the production loop works.

Working principle

The factory should receive one packet with everything it needs to print and ship. If a person has to interpret a screenshot, redraw the design, or search through Shopify for context, the system has not reached MVP.

The system has two centers. Shopify handles commerce. The PäSTE app handles the custom work around the order: design state, asset approval, rendering, factory handoff, and royalty accounting.

Component split

Shopify

Commerce system

Shopify owns products, variants, cart, checkout, taxes, shipping, customer records, payment status, and fulfillment status. PäSTE passes a design ID and preview into the cart line item.

PäSTE designer

Design surface

The designer owns the collage interaction: front and back templates, approved assets, drag, scale, rotate, layer order, preview, and saved design JSON.

PäSTE backend

Order orchestrator

The backend saves designs, receives Shopify webhooks, locks the design to the paid order, generates production packets, tracks status, and writes royalty ledger rows.

Renderer

Proof and print files

The renderer outputs customer previews, QC proofs, and print-ready files. Exact output depends on the factory's required format, DPI, artboard size, color profile, and bleed.

Factory packet

One paid line item, one packet

Each packet includes Shopify order ID, line item ID, blank SKU, garment color, size, quantity, shipping address, front and back proofs, front and back print files, asset manifest, label instructions, packaging instructions, QC image, notes, and tracking return path.

Royalty ledger

Accounting before automation

The ledger records IP holder, guest designer, affiliate, order value, refund status, eligibility thresholds, and payout status. Automated payout can wait; ledger accuracy should come first.

The first build should be small enough to ship and complete enough to test the business. The goal is one real operating loop. Future surfaces can follow after the loop works.

Three-phase MVP

Phase 1

Confirm production specs

Before the interface is finalized, the factory requirements need to be known: file format, DPI, artboard size, color profile, bleed, print area, garment color rules, order intake method, status updates, and tracking return method.

Output: a factory data contract.

Phase 2

Build the working loop

Build one IP page, one approved asset pack, one garment workflow, front and back design surfaces, saved design JSON, Shopify cart handoff, paid order webhook, and automatic production packet generation.

Output: a test order that produces a real packet.

Phase 3

Operate the first orders

Send packets through the factory's preferred channel. Track exceptions. Return tracking to Shopify. Calculate royalty rows. Document the operator workflow while the first orders expose what needs to be tightened.

Output: an operator-ready v1 process.

Questions to confirm

Production

Who is the production partner, what exact files do they need, what does zero-dimensional printing mean in their workflow, and what does the 100-shirt minimum apply to?

Shopify

Does the store already exist, which plan is expected, and should v1 be a PäSTE-owned storefront before any partner-site embed work begins?

IP

What is the first real IP partner, who approves the asset pack, and can users use only approved assets in v1?

Royalties

Are royalties calculated on gross retail, net merchandise revenue, or post-discount revenue? Are the first three designer and affiliate sales unpaid forever or paid retroactively once the threshold is reached?

Role

Is Noah being asked to build the first system, operate it day to day, train an operator, or hold a technical product lead role after launch? The answer affects scope, economics, and pace.
