Prepared for John and Unitus · Private proposal · June 2026

Unitus SEO and traffic-readiness review.

We ran the first layer of the review using the access already available: the draft Shopify redesign, a read-only Shopify inventory, and the storefront pages a search engine can read. The next layer connects that work to real search and visitor data.

The short version

The redesign is close enough to evaluate

The current evidence points to a practical sequence: clean up the Shopify theme, catalog, landing paths, and measurement baseline before deciding whether Unitus needs a larger build change.

That gives the work two layers. The first layer finds the traffic-readiness issues visible now. The second layer uses GA4, Search Console, and aggregate Shopify data to decide what matters most commercially.

No platform changes were made in the first layer. The review was read-only.

Layer one

What we can already see

01

The launch story needs one source of truth

Sale language, seasonal language, collection roles, and product state need to agree before search, email, SMS, and paid traffic all point at the same storefront.

02

Navigation and landing paths need sharper intent

Some calls to action and navigation routes send shoppers to broad collection views. SEO and campaign traffic work better when each link lands on a page with a clear job.

03

Product pages need stronger page structure

The product template appears to have a semantic heading issue. That affects accessibility and the way search engines understand the main product topic.

04

Metadata and image context need a priority pass

Important pages need custom titles and descriptions, and product media needs descriptive context. That supports snippets, image search, accessibility, and AI shopping surfaces.

05

The catalog model should be tightened before traffic scales

Product types, options, SKU discipline, and collection roles all feed filters, search, campaign segmentation, feeds, and reporting. Small inconsistencies compound once the store is live.

06

Measurement is the current ceiling

The first layer identifies structural issues with confidence. Access-backed data ranks the work by search demand, traffic value, conversion paths, and revenue impact.

Layer two

What read-only access would add

The second layer should answer different questions than the first. It should tell us where demand already exists, which pages are closest to moving, and which fixes deserve the first implementation slot.

Search Console

The queries, impressions, clicks, and pages Google already associates with Unitus. This shows page-two opportunities, weak click-through snippets, indexation problems, and overlap between pages.

GA4

Where visitors land, which channels bring them in, how mobile compares with desktop, and which paths lead toward product views, cart, and purchase.

Shopify

Aggregate product, collection, channel, and discount reporting to rank SEO work by commercial importance. Customer and order rows would stay out of scope unless a separate reason and approval exist.

Feed surfaces

Merchant Center or feed diagnostics, if available, to see whether Shopify product data issues also appear in Google Shopping and product-discovery surfaces.

Current read-only status

The second layer is now evidence-backed

Search Console and Shopify read-only access are working. The first source-backed pass found a concrete metadata queue without needing customer rows, order rows, or platform edits.

Search demand

Search Console shows 7,919 clicks and 178,169 impressions over the last 90 days, with a 4.44% click-through rate and average position 6.3 on the URL-prefix property.

Priority URLs

Sixty pages with at least 250 impressions were reviewed. Thirty surfaced as priority rows because of click-through leakage, page-two opportunity, zero-click visibility, or page-type mismatch.

Metadata queue

Twenty-three collection rows, eight home or page rows, and the highest-priority product rows have now been mapped. A first-pass packet is ready for nine resources with missing SEO fields, but no Shopify write has been approved or made.

Open access gaps

GA4 API access still needs the Analytics read-only delegation scope. Merchant diagnostics still need the relevant Google APIs enabled before item-level feed work can be ranked against the SEO queue.

Build approach

The platform question belongs beside the audit

The right question is whether the current Shopify build gives Unitus enough control for SEO, structured data, landing pages, performance, and measurement.

The evidence so far favors a Shopify-first cleanup: theme semantics, collection roles, product metadata, image context, taxonomy, and measurement. A headless or hybrid build only belongs on the table if we find a clear ceiling that Shopify theme work, Online Store 2.0, sections, metafields, metaobjects, or light middleware cannot solve cleanly.

Theme cleanup

Fix page structure, headings, metadata patterns, schema details, internal links, and app weight inside the current Shopify path.

Native Shopify

Use sections, metafields, metaobjects, Search and Discovery, better templates, and a controlled content model before proposing a different storefront architecture.

Hybrid or headless

Evaluate only if the data proves that Unitus needs landing pages, performance control, experimentation, or content systems the current Shopify path cannot support without fragility.

Proposed work

What we would do next

Step 01

Set the access boundary

Confirm read-only GA4 and Search Console access, plus aggregate Shopify reports if available. Keep customer rows, order rows, platform edits, and campaign sends out of scope unless separately approved.

Step 02

Build the baseline

Map current search demand, landing pages, channel paths, indexed URLs, priority collections, product families, and measurement gaps.

Step 03

Turn the audit into a queue

Separate quick fixes from theme work, catalog operations, content work, measurement setup, and platform constraints. Each item gets an owner, a reason, and a clear point where Unitus approves any change.

Step 04

Decide the build path

Make a clear recommendation: current theme cleanup, Shopify-native enhancement, a small middleware/tooling layer, or a deeper build investigation if the evidence justifies it.

What we need

A short access list

  • GA4 read-only access.
  • Google Search Console read-only access.
  • Aggregate Shopify reports or admin summaries if they are easy to share.
  • Merchant Center or product-feed diagnostics if Unitus uses them.
  • The current Shopify implementation owner and whether the build team is still active.

The first pass should stay aggregate and view-only. If a later question requires protected customer or order data, that becomes a separate decision.

The next step

Start with the second layer

If this direction is useful, the next step is read-only access and a short call to agree on the first set of priorities.
