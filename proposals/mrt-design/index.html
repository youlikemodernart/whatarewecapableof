Spec and Capture Pilot.

A focused first build for MRT's spec material, field notes, comment responses, and project follow-through. The pilot turns material the team already has into source-linked workflows MRT can review and keep.

Recommendation

Start with the work already in motion

I went back through the recording from our call. My recommendation is to start with the Spec and Capture Pilot.

The useful starting point is material MRT already has: spec books, product standards, old project sections, site notes, comment responses, and the decisions that get made in the field or during client calls.

The goal is a shared workflow your team can use and review, with MRT keeping professional judgment at each step.

Why start here

01

MRT is already using AI on individual tasks. The next step is making that work reusable across the office.

02

A spec and capture pilot creates value while keeping direct Revit automation, accounting access, and broad client-call recording outside the first build.

03

The pilot creates a safer base for later Revit or construction-document automation because the system starts with the standards, comments, decisions, and project context those tools would need.

Pilot scope

What the pilot produces

Workflow 1

Spec standards workflow

We use a small set of MRT's existing spec material to build a repeatable spec-update workflow.

Outputs: intake of 2-3 representative spec books or templates, a current-state map, a vendor and product standards register, one working prototype, and a review checklist.

Workflow 2

Capture-to-checklist workflow

We build a lightweight way to turn field notes, meeting notes, voice memos, note photos, and comment responses into organized project records and action lists.

Outputs: post-call or post-site voice memo workflow, note-photo intake, site report template, project decision log, follow-through checklist, and city-comment or client-ask tracker.

Workflow 3

Tool path recommendation

Some of this may belong in tools MRT already has or can buy. CompanyCam came up on the call, and Teams, Outlook, and SharePoint appear to be the more natural environment for the office.

Outputs: recommendation for the site-reporting layer, recommendation for where spec standards and project records should live, and a short list of what to automate now, what stays human-reviewed, and what waits.

How we would run it

Step 1

Setup

MRT sends a few examples: 2-3 spec manuals or templates, one recent updated spec section, reused product or vendor notes, one active project, and sample notes, photos, comments, reports, or meeting records from that project.

Step 2

Build

We organize the source material, identify the repeatable parts, and build the first version of the spec and capture workflows. The first version can use exported files if direct access is heavier than the pilot needs.

Step 3

Review

Omar, Taz, or the right MRT reviewer tests the outputs against real project material. We tighten the workflow based on that review and produce a handoff packet your team can keep using.

Guardrails

01

Professional approval stays with MRT.

MRT's appropriate reviewer approves any specification, drawing response, project record, or client-facing output before it is used, issued, submitted, or relied on.

02

The pilot stays outside sensitive writes.

Accounting and Intuit time-entry writes stay out of scope. Direct Revit/CD automation waits for a separate feasibility path with project files, templates, and review rules.

03

Client-call recording remains a policy choice.

Client-call recording should expand only after MRT chooses consent language and retention rules that fit its client relationships.

What success looks like

At the end of the pilot, MRT should have a working spec-update process, a shared product/vendor standards structure, a capture workflow, a reviewable checklist, and a clear recommendation on what to automate next.
