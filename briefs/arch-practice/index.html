Architecture Firm Outline

Yes. The right move is to treat an architecture office the same way we treated publishing and gallery: first map the firm's recurring work, then turn each stable role into an agent with a leash, sources, artifacts, and approval gates.

Core mapping pattern

An architecture office has one workflow spine:

lead→fit check→proposal→contract→programming→schematic design→design development→construction documents→permit/bid + construction administration→closeout

Each phase creates recurring work objects:

  • lead record
  • client brief
  • proposal and fee model
  • contract and scope matrix
  • project schedule
  • consultant roster
  • design decision log
  • code and zoning checklist
  • Revit/BIM model or drawing set
  • permit package
  • RFI/submittal log
  • field report
  • punch list
  • project profitability and lessons-learned record

Paperclip maps well because every one of those can become an issue tree.

Autonomy gradient

Tier 1: high autonomy, light review

Rule-bound, checkable work.

  • meeting notes and action logs
  • CRM cleanup
  • project folders and naming
  • timesheet reminders
  • budget dashboards
  • drawing index checks
  • permit checklist assembly
  • RFI and submittal log maintenance
  • product data collection
  • template proposals
  • closeout archive packets

Tier 2: assisted, human decides

Judgment-adjacent work.

  • project-fit analysis
  • fee and staffing scenarios
  • programming synthesis
  • precedent research
  • code interpretation drafts
  • consultant scope comparison
  • SD/DD/CD phase readiness checks
  • design option summaries
  • material or assembly comparisons
  • permit response drafts
  • proposal positioning

Tier 3: human-led

The firm's moat and liability core.

  • design authorship
  • client relationship
  • final scope, fee, and contract terms
  • consultant selection
  • sealed documents
  • code compliance sign-off
  • construction observation judgment
  • pricing and risk decisions
  • hiring and firing
  • external submissions and sends
  • anything that creates professional liability

Where this fits in the skill system

I would make this a standalone, crosslinked skill, probably:

architecture-practice or architecture-office-operations

It should not be buried inside arch-construction-docs. That skill is about producing architectural documents. This new layer would be about running the practice.

Crosslinks:

  • arch-construction-docs: CD production, drawing sets, BIM/CAD standards, QA, code documentation.
  • construction-technology: delivery systems, IPD, consultants, BIM/VDC, construction team structure.
  • residential-architecture-construction: residential code and building scope.
  • business-consulting: expert-firm economics, positioning, service operations, org design, founder bottleneck.
  • contract-writing: scope, proposals, AIA-style agreement review support.
  • bookkeeping: invoicing, WIP, project profitability, monthly close.
  • paperclip-platform: runtime pattern, company, agents, issue tree, gates, audit log.

Thin areas to research before this becomes strong

Our current archive gives us a seed. The missing substrate is architecture-practice management:

  1. Firm economics: fees, utilization, project profitability, WIP, cash flow, overhead, staffing ratios.
  2. Sales and positioning: how architects get work, referrals, competitions, developer relationships, niche formation.
  3. Proposal and contract structure: scope boundaries, additional services, exclusions, standard agreements, liability.
  4. Project management: phase gates, workplans, deliverables matrix, consultant schedule, budget-to-actual.
  5. CA and liability: RFIs, submittals, site observations, change orders, field reports, punch lists.
  6. Permitting and entitlement: zoning, planning, building department workflow, expediters.
  7. BIM management: model standards, LOD, coordination, template libraries, QA protocols.
  8. Career ladder and staffing: principal, associate, project architect, designer, job captain, BIM manager, interns.
  9. Consultant ecosystem: structural, MEP, civil, landscape, code, envelope, cost estimator, renderer.
  10. Operating partner split: what a non-architect can own without blurring licensure, authorship, or liability.

Recommended build path

  1. Create the architecture office map: role chart, workflow spine, trust gradient, external gates.
  2. Do role deep dives: one memo per role with recurring work, artifacts, metrics, sources, failure modes, AI fit.
  3. Build the minimal Paperclip company: Principal, Operating Partner, Project Manager, Proposal/Fee, BIM/CD QA, Finance.
  4. Add technical agents only after the operating spine works: Code, Specs, CA, Consultant Coordination.
  5. Keep design and liability human by default.

The key principle: agents should run the apparatus around the architect, not replace the architect. The valuable firm is a human design practice with an unusually disciplined operating system.
