The MRT operating brain.

A larger project for giving MRT one shared AI-assisted layer across specs, project records, standards, capture, comments, and read-only drawing/model references.

Separate larger path

What this proposal covers

The spec and capture/QC pilot is the right first build. It is narrow enough to test quickly and broad enough to touch the repeated work MRT described on our call.

There is also a larger version of the work: an MRT operating brain.

This would be a longer project. The goal is to give MRT one shared AI-assisted layer that helps the team find what was decided, what still needs review, where a comment came from, which standard applies, and what project context should be in front of someone before the next move.

Read the first pilot proposal

What an operating brain is

01

A private system next to the tools MRT already uses: Revit, Teams, Outlook, SharePoint, Intuit, project folders, and the judgment of the architect.

02

A source-linked answer layer. If someone asks a question, the answer points back to the spec section, meeting note, site record, comment, sheet, schedule, or exported model data behind it.

03

A way to preserve useful individual AI experiments as office-level knowledge when they are worth keeping.

Source layer

What it could read

Standards

Spec books, templates, and office standards

The system starts with approved material MRT already reuses: specs, preferred products, vendor notes, baseline assemblies, and review rules.

Records

Meeting records, field notes, and site reports

Approved notes, voice memos, site photos, reports, and meeting records become project memory instead of scattered context.

Comments

Permit comments, client comments, and response letters

Comment packets become traceable records with owner, source, response, drawing/spec reference, and review status.

Drawings

PDF sets and Revit-derived exports

Approved drawing sets, schedules, sheet indexes, view exports, and model reports become read-only reference material the system can search and cite.

Structure

Project folders and naming patterns

The system learns how MRT already organizes work so the output fits the office instead of forcing a new filing model too early.

Revit boundary

The Revit part should start as a read/reference layer.

Read
sheets, schedules, PDFs, view exports, parameters, and reports
Cite
visible drawing and model-derived references behind an answer
Flag
areas a reviewer should check
Exclude
model edits, final CDs, submissions, and professional approval

Direct Revit automation should become its own technical and professional-review project if it becomes useful later.

System shape

What the system would do

Layer 1

Build the source map

Map where MRT's project knowledge lives today: Teams, Outlook, SharePoint, local/server folders, Revit files, spec folders, comment logs, field notes, and project numbers.

Output: a clear source map and access plan.

Layer 2

Create the standards layer

Hold MRT's repeatable standards: spec sections, preferred products, vendor notes, baseline assemblies, office defaults, and review rules.

Output: a standards base that starts with the first pilot and expands into an office resource.

Layer 3

Create the project memory layer

Collect approved meeting notes, field notes, comments, decisions, and project records into one source-linked project memory.

Output: less re-explaining and easier decision tracing.

Layer 4

Create the capture layer

Turn voice memos, note photos, Teams records, site photos, and comment packets into organized records, action lists, and review checklists.

Layer 5

Create the QC and comment layer

Track permit comments, client asks, and field decisions with owner, source, response, drawing/spec reference, and review status.

Output: human review support, with no claim that a drawing set is complete.

Layer 6

Create the Revit reference layer

Use approved drawing sets, schedules, and Revit-derived exports as searchable, citable reference material.

Output: a better read on the project, with model editing outside this scope.

How we would build it

Stage 1

Source map and rules

Identify the systems, file types, permissions, and review boundaries. Choose one active project and one or two repeatable workflows to prove the system.

Stage 2

First brain prototype

Build the first private prototype around the selected project. The first version should answer source-linked questions, organize project records, and produce reviewable checklists.

Stage 3

MRT review loop

Omar, Taz, or the right project lead tests the system against real work. We revise the records, prompts, checklists, and source rules until the output fits the way MRT works.

Stage 4

Office expansion

After the first project works, expand across more project types, more source folders, more standards, and more reviewers.

What success looks like

A useful operating brain gives project leads a faster way to get current before a meeting, site visit, or review.

01

Ask what was decided on a project and see the source behind the answer.

02

Turn a site note or voice memo into a project record and action list.

03

Keep product and spec standards shared across the office.

04

Track comments and drawing/spec follow-through without keeping three screens open.

05

Use Revit-derived material as a reference source while keeping the model protected.

Relationship to the pilot

The first build should earn the larger one

The spec and capture/QC pilot is the first slice of this larger system. It tests the parts that matter most: source intake, standards, capture, checklists, and MRT review.

If the pilot works, the operating brain is the path for turning that first slice into a broader office layer.

If the pilot stalls, the larger brain waits until the workflow is worth expanding.
