# Proposal composition system

Created 2026-04-24.

This system governs how visual rhythm enters What are we capable of? proposal pages. It is built for long, text-heavy proposal sites whose final form should follow the argument rather than a rigid template.

The repeatable part is the thinking: read the proposal, map the argument, identify where the reader needs proof or air, then show the proposed interruptions in an in-flow placeholder draft before final media is chosen.

## Pipeline position

Default sequence: make the proposal first, then run a visual composition pass. Let the text, findings, research structure, and scope take shape before deciding where images belong. This keeps proposals from becoming templates and prevents available imagery from controlling the argument.

Exception: if the proposal's subject is inherently creative or media-led, the visual system can shape the proposal from the beginning. Compassion is the model: the form of the proposal depended on showing content direction, feed logic, and visual examples. In those cases, the proposal production pipeline should mark the work as creative-led before drafting so images, specimens, video, or mockups become part of the core argument rather than a post-pass.

For the ordinary consulting proposal, treat images as a second pass:

1. Draft and stabilize the text-first proposal.
2. Read it as an argument and identify density, proof, atmosphere, and fatigue points.
3. Create an in-flow slot draft using the real proposal copy.
4. Curate or generate media for the approved slots.
5. Implement only the images that materially improve trust, comprehension, or cadence.

## Purpose

The proposal pages are doing several jobs at once:

- show that we paid attention
- make outside research legible
- help a recipient recognize their own context
- convert observations into a possible engagement
- give the reader enough confidence to take a conversation seriously

A page full of prose can do this, but the reader pays a cost. Dense paragraphs ask for sustained attention before trust has fully formed. Visual material can lower that cost when it functions as evidence, comparison, artifact, sequence, or pause.

## Relationship to the existing design system

The proposal composition system sits inside the locked whatarewecapableof design language:

- Typography carries the page.
- Images and video enter as content, never as chrome.
- The page stays left anchored.
- The baseline grid stays intact.
- Black text on white remains the default.
- Blue remains a state color, not a decorative accent.
- Visual hierarchy comes from position, grouping, scale restraint, and content order.
- No cards, shadows, hero imagery, decorative icons, gradients, or template polish.

The proposal pages are allowed to become denser than the public site because the proposal is a work object. The density still needs rhythm. A reader should feel carried through the argument rather than dropped into a wall of evidence.

## Core principle

Every visual insertion needs a job, but the job can be atmosphere.

Before adding any image, video, diagram, screenshot, or specimen, write one sentence that explains what it does for the reader. If that sentence is weak, cut the insertion. The sentence does not need to prove the adjacent paragraph. Sometimes the right job is to give the reader air, mark the client's product world, or reset the cadence before the next finding.

Good rationale examples:

- Shows the current experience so the reader does not need to imagine it.
- Places the client's current feed beside the proposed direction.
- Turns an abstract system into a visible workflow.
- Gives proof for a claim that would otherwise sound speculative.
- Creates a pause before the proposal shifts from research to scope.

Weak rationale examples:

- Breaks up the page.
- Adds visual interest.
- Makes it feel designed.
- Uses the images we already have.

Breaking up the page is a symptom-level goal. The deeper question is why the reader needs a break at that point.

## Proposal beat map

Start every composition pass by mapping the proposal into rhetorical beats. The beat map identifies what the reader is being asked to do at each point.

| Beat | Reader task | Common content | Visual opportunity |
|---|---|---|---|
| Orientation | Understand who this is for and what kind of document this is | Meta line, intro paragraph, tab guide | Usually none. Keep fast and quiet. |
| Recognition | See their own world accurately described | Company context, known facts, current constraints | Light evidence only if the recipient may doubt the frame. |
| Observation | Consider what we noticed | Findings, public data, site behavior, social patterns | Strong: screenshot, source excerpt, stat field, annotated proof. |
| Evidence | Trust that the observation is grounded | Tables, screenshots, scraped data, source links, quotes | Strongest zone for media. |
| Interpretation | Understand what the evidence might mean | Consequences, patterns, hypotheses | Diagram, comparison, sequence, or quiet typographic emphasis. |
| Possibility | Imagine what could change | Concepts, content directions, systems, programs | Specimen, artifact mockup, process strip, video walkthrough. |
| Offer | Understand how we would work | Phases, tiers, deliverables, division of labor | Timeline, system diagram, deliverable artifact. |
| Decision | Know what the next conversation should confirm | Open questions, call to action | Usually minimal. Let the prose land. |

A proposal does not need every beat. The map is diagnostic. It prevents media from clustering only where assets happen to exist.

## Density and fatigue signals

A visual interruption is worth considering when any of these appear:

- three or more long paragraphs with no structural change
- repeated finding blocks with similar length and cadence
- a table that carries the whole argument without a visual summary
- an abstract system described in prose for more than one paragraph
- a claim that depends on a visible public artifact
- a before and after argument made without a before and after image
- a process described as if it is linear when it may be spatial or cyclical
- a section transition where the reader needs to shift from evidence to possibility
- mobile reading where the proposal becomes a long stack of undifferentiated text

A visual interruption is usually wrong when:

- the paragraph is short and already clear
- the prose is building momentum toward a conclusion
- the image repeats exactly what the sentence says
- the asset quality is weaker than the claim
- the insertion makes the recipient look bad without adding value
- the media would be more interesting than the point it supports

## Media jobs

Use media jobs before layout moves. The job explains why the object exists.

| Job | Best media | Use when | Avoid when |
|---|---|---|---|
| Proof | Screenshot, source excerpt, public page capture, data snippet | A claim needs visible grounding | The proof is obvious or too low quality |
| Comparison | Diptych, feed comparison, before and after pair | The argument depends on contrast | The two sides are not meaningfully comparable |
| Artifact | Mocked deliverable, document sample, dashboard, line sheet, catalog view | The reader needs to see what they would receive | The artifact would overpromise detail |
| Atmosphere | Client-world image, field image, environmental clip | The proposal needs a sense of place or audience | The image becomes decoration |
| Sequence | Process strip, 3 to 6 frames, short video, stepped diagram | Change over time matters | The sequence can be explained in one sentence |
| Diagram | System map, workflow, relationship model, data flow | Relationships matter more than surfaces | The diagram would become generic consulting theater |
| Specimen | Concrete content example, post mockup, campaign format, offer sample | The proposal needs examples, not just categories | The specimen reads as final creative direction too early |
| Pause | Typographic field, quiet stat field, single image with caption | The reader needs air before a major shift | The pause interrupts a necessary chain of reasoning |

## Composition moves

These are moves, not templates. A proposal should use only the moves its argument earns.

### Evidence plate

A single image, screenshot, source excerpt, or video with a mono label and a short caption.

Use for:

- public page screenshots
- analytics stack evidence
- cause page examples
- product page captures
- social feed examples
- platform screenshots

Default treatment:

- full proposal column or wide rightward overflow
- caption below
- mono label above or inside caption
- no border unless the edge needs definition
- no decorative background except a pale surface for transparent assets

### Diptych

Two media objects placed side by side on desktop, stacked on mobile.

Use for:

- current versus proposed
- client versus competitor
- visible page versus hidden infrastructure
- old system versus new system
- before versus after

Rules:

- The comparison must be named in the caption.
- Use equal visual weight unless one side is intentionally subordinate.
- Do not use if one asset is much weaker than the other.

### Specimen grid

A set of concrete examples, each with number, label, media, description, and role.

Use for:

- content pillars
- social post formats
- deliverable examples
- campus event types
- private catalog examples

Rules:

- Keep descriptions short.
- Three to six specimens is the useful range.
- The grid should demonstrate the system, not exhaust it.

### Process strip

A horizontal or vertical sequence of steps, frames, or states.

Use for:

- how a buyer moves through a flow
- how a student club launches
- how a collector record becomes outreach
- how raw research becomes a proposal
- how content moves from church partner to published post

Rules:

- Each step needs one verb.
- Keep the sequence short enough to remember.
- Use horizontal scroll only when the sequence benefits from lateral movement.

### System diagram

A minimal text-first diagram showing relationships between entities.

Use for:

- data models
- ownership flows
- audience distribution
- stack architecture
- approval paths
- operating rhythms

Rules:

- Prefer words and lines to icons.
- Keep the diagram readable without a legend.
- If it needs a legend, it may be too complex for the proposal page.

### Document excerpt

A source quote, scrape, table fragment, or document-like proof object.

Use for:

- public claims
- mission language
- product catalog facts
- exact platform copy
- evidence that should be read as primary source material

Rules:

- Excerpt only the necessary lines.
- Cite the source in the caption.
- Avoid making the client feel exposed unless exposure is central to the proposal.

### Quiet stat field

A sparse group of numbers with enough space to reset the reader's attention.

Use for:

- scale
- contradiction
- goals versus budget
- before a findings section
- before an investment section

Rules:

- Four to six numbers maximum.
- Each number needs a label that can be understood without the surrounding paragraph.
- If the numbers need explanation, pair the field with a short paragraph after it.

### Video plate

A video treated as a piece of evidence or demonstration, with poster, controls, caption, and transcript or captions when spoken.

Use for:

- walkthroughs
- screen recordings
- social content examples
- process demonstrations
- recorded voice or field material

Rules:

- No autoplay for meaningful video.
- Use `preload="metadata"`.
- Provide poster imagery.
- Provide captions or a transcript when speech matters.
- Keep the player surface quiet.

## Video hosting and serving

Video can fit this system if it is treated as a restrained artifact.

### Default hosting choices

| Option | Use when | Tradeoff |
|---|---|---|
| Repo-hosted MP4 or WebM | Short clips, screen recordings, small files | Simple but can bloat the repo |
| Vercel Blob, S3, or Cloudflare R2 | Larger proposal media or reusable video assets | Requires storage setup and asset workflow |
| Vimeo or YouTube | Long or externally shared video | Embedded chrome may fight the proposal surface |

Default recommendation: repo-host very short clips during prototyping. Move to object storage once video becomes a normal proposal material.

### Video treatment

The video plate should look closer to a source document than to a media player. The visible elements are:

- poster frame
- native controls unless custom controls are justified
- mono label
- short caption
- optional transcript link or inline transcript

Custom controls should use text, not icons: `PLAY`, `PAUSE`, duration, transcript. Build custom controls only if native browser controls feel too visually loud during review.

### Accessibility and performance

- Spoken video needs captions or transcript.
- Do not communicate essential meaning through motion alone.
- Set explicit width and height attributes.
- Keep `preload="metadata"` unless the video is the central above-fold object.
- For looping ambient clips, include a pause mechanism and respect reduced motion.
- Compress video before upload. Favor 720p or 1080p depending on the actual display size.

## Selection process

A composition partner should follow this order every time.

### Phase 1: Read

Read the proposal page, proposal tracker entry, project notes, and any available client research. Do not start with assets. Start with the argument.

### Phase 2: Map

Create the beat map. Mark each section with:

- beat type
- reader task
- density level: low, medium, high
- trust requirement: low, medium, high
- possible media jobs
- visual risk

### Phase 3: Inventory

List available media and possible media:

- existing images in the project
- screenshots that can be captured
- public pages worth quoting
- diagrams that can be drawn from known facts
- short videos that could be recorded
- artifacts that could be mocked up honestly

Classify each asset by job, quality, and confidence.

### Phase 4: Propose

Recommend insertions with exact placement anchors. Default to a single body-width image or simple video unless the argument clearly needs another structure.

Default frame logic:

- `16:9` for the first pass
- `4:3` or `7:5` when the image should feel less cinematic and more object-like
- `5:7` when the surrounding context wants a vertical image
- diptychs, grids, and tall paired screenshots only when Noah asks for them or the proposal clearly needs comparison, sequence, or specimens

Each recommendation needs:

- section or line anchor
- media job
- frame ratio
- asset source or asset needed
- rationale
- mobile treatment
- implementation complexity
- risk
- priority

### Phase 5: Cut

Cut aggressively. A good proposal may need three visual insertions, not ten. Prioritize the insertions that increase trust, clarify the offer, or give the reader needed air.

### Phase 6: Draft placeholders

Before final images are inserted, create an in-flow placeholder draft when Noah is collecting imagery. The draft should use the actual proposal copy and grey frames at the proposed insertion points. The frame itself carries the slot ID, ratio, suggested content, and Are.na tag.

### Phase 7: Implement

Implement final media only after the placeholder draft is approved or the task explicitly asks for implementation. Use reusable classes where possible. Keep page-specific CSS small.

### Phase 8: Review

Review at 375px, 768px, 1280px, and 1600px. For each insertion, ask:

- Does it change what the reader understands?
- Does it appear at the right moment in the argument?
- Does it preserve the site register?
- Does it make the section easier to read on mobile?
- Does it introduce performance, accessibility, or path fragility?

## Local slot boards

A slot board should usually be an in-flow composition draft. It is a local copy of the proposal with grey placeholder frames inserted exactly where the images would live. The goal is to let Noah judge cadence in the real page format before collecting, cropping, or inserting final images.

Default location while developing:

```txt
proposals/<slug>/slots/index.html
```

These boards are local-only unless Noah approves deployment. The repo ignores `proposals/*/slots/` to prevent accidental publishing. To preview a board, serve the project root with `python3 -m http.server 8888` and open the local URL.

Each placeholder frame should show the slot ID, default ratio, curation suggestion, and Are.na tag. The page should default to a single image frame, not a grid of all possible slots.

When draft images are available, use non-destructive CSS cropping first. Keep raw source files under `proposals/<slug>/img/src/`, which is ignored. Noah may manually crop a source image, overwrite the local file, and ask for a re-preview. Final deployable crops should later move to `proposals/<slug>/img/`.

Filled draft images should not have visible borders or frames unless a specific evidence object needs edge definition.

## Anti-patterns

- Filling every section with a visual.
- Using images as decoration after dense prose instead of choosing the right interruption point.
- Making proposal pages look like case study templates.
- Letting the media become more polished than the offer.
- Mocking artifacts so specifically that the recipient thinks the work is already decided.
- Using screenshots that embarrass the client without giving them a path forward.
- Adding video because video feels premium.
- Adding diagrams that say less than the paragraphs they replace.
- Treating mobile as a stacked afterthought.

## Definition of success

The reader moves through a long proposal with less fatigue and more trust. They understand the evidence faster. They can picture the work without feeling that the proposal has become a generic deck. The visual system remains quiet enough that the content feels discovered, not packaged.
