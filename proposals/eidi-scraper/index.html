Eidi Deal Source Monitor.

A small first build that keeps an eye on the deal sources Eidi already checks, then sends a short digest when something new or changed deserves a look.

Direction

Shift Phase One toward the places Eidi already looks

After the latest conversation, Phase One should move from lease abstracts to deal-source monitoring. If the current ChatGPT workflow is covering lease questions well enough, the better first test is a monitor around the websites and pages Eidi already uses to find opportunities.

Eidi brings the first source list. We check what each site allows, confirm what should stay manual, and build around the sources approved for monitoring. The output is a short digest of new items, changed listings, and opportunities that need a human look.

The earlier lease-abstract packet should stay paused while this direction is reviewed.

The first build

The first version should stay narrow on purpose. It should reduce repeat checking, keep source handling clean, and give the team a better starting point for review.

Input

First source list

Eidi sends the listing sites, broker pages, special-servicer pages, relationship pages, and other sources the team already watches.

We do not guess the list. We start with the places the team recognizes.

Review

Source-by-source rules

Before a source is used, we sort it by access type, what the site allows, how often it changes, and what information is visible.

Some sources can be watched directly. Some should stay manual. Some may need permission, subscription access, or a separate workflow.

Watch

Approved sources only

The monitor checks agreed fields from agreed sources on an agreed cadence.

Login-based and subscription sources wait until Eidi approves handling for that specific source.

Digest

Short review output

The digest shows new opportunities, changed listings, source issues, likely-fit items, and items that need a human check.

The first output should be easy to read, easy to forward internally, and easy to mark up with team notes.

Tune

Better with each pass

Eidi marks each item useful, irrelevant, duplicate, stale, or needs more context.

Those notes sharpen the filter and show which sources are worth watching continuously.

What the digest can show

The field list should follow how Eidi buys. A website may show more, but the digest should only pull forward what helps the team decide whether to review the opportunity.

FieldUse
Source and linkShows where the opportunity came from and gives the team a path back to the listing.
Property or portfolioNames the asset, portfolio, location, and asset type when listed.
First-screen factsCaptures asking amount if listed, square footage or acreage, occupancy notes, and contact details where available.
Change notesShows when an item first appeared and whether asking amount, status, availability, or source format changed.
Review noteFlags why an item may fit, why it may not, and what information is missing.

Phase One

Setup and first digest cycle

Phase One should answer a practical question: does this save time and point the team toward better things to review?

Step 1

Source inventory

Eidi sends the first source list. We sort each source by access type, allowed use, expected change frequency, and visible fields.

Step 2

Buying criteria

One working session defines the first screen: markets, asset types, deal size, portfolio preference, disqualifiers, and fields the team wants in the digest.

Step 3

Monitor setup

Build the monitor around approved public sources and any other sources Eidi explicitly approves for this first cycle.

Step 4

Digest draft

Create an email-style digest that can run daily, weekly, or twice weekly, depending on Eidi's preference.

Step 5

Review and tune

Run the first cycle, review what was useful, then adjust the source list, fields, and filters before expanding.

Guardrails

These keep the first build focused and safe.

Sources

Eidi starts the list.

We can suggest additions later. The first version uses sources the team already recognizes.

Rules

Each source is checked before it is watched.

Public, login-based, subscription, and relationship-sensitive sources need different handling.

Systems

No operational-system dependency.

Phase One should not require Yardi access, tenant data, lease files, inbox access, or production credentials.

Judgment

The digest narrows the review list.

The monitor helps reduce the pile. Eidi still decides what to pursue.

What success looks like

At the end of Phase One, Eidi should have five practical answers.

1
Did the monitor catch items the team would otherwise check manually?
2
Did the digest save review time?
3
Did the filtering match Eidi's deal judgment?
4
Which sources are worth watching continuously?
5
What should the next layer be?

If the answers are strong, the monitor can become an ongoing service. If the answers are mixed, we adjust the sources, fields, or cadence before expanding.
